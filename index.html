Broken Authentication



          public updatePassword() {
            $sql = "UPDATE customer SET password = ?, changed = NOW() WHERE id = ?";
            $stmt = $db->prepare($sql);
            $stmt->bind_param('si', $_POST['password'], $this->customerId);
            $stmt->execute();
          }

What does it mean?

when you ...

  • store passwords unencrypted/not hashed
  • expose session IDs (in URLs?, to JS in Cookies?)
  • don't rotate session IDs after login (session fixation)
  • don't timeout/invalidate session IDs or use weak IDs
  • transmit authentication not over https

Mitigation

  • sessionId storage: secure, httpOnly, sameSite and in cookie only
  • use known and tested flows (OAuth, OpenID, ...)
  • detect/prevent brute forcing
  • renew the sessionID after any privilege level change
  • set a short session timeout (idle, absolute, renewal)
  • provide a logout button
  • logging sessions lifecycle
  • btw: manage SessionIds just like any other user input

Sensitive Data Exposure

What data exposed to ...?

  • Usernames (or emails, passwords, ...)
  • Creditcard, banking infos, ...
  • Other systems hostnames, usernames, passwords
  • Backup Files of the application
  • Logfiles, System output, ...
  • Session IDs, Tokens ...
- even hashed passwords can be retrieved with rainbow tables [Rainbow Tables](https://crackstation.net/)

How to prevent this?

think before you store!

  • TLS encrypt connections
  • webroot ... must not contain backups, logs, ...
  • hash passwords
  • protect sessions (app & webserver cfg)
  • especially Credit Card/Banking Data - don't store it (yourself)
  • scan with OWASP Zed Attack Proxy, Nikto, W3AF, ...
  • scan regularily, before each deploy and on PROD

          <foo>&xxe;</foo>

          <!ELEMENT foo ANY >
          <!ENTITY xxe SYSTEM "file:///dev/random" >]>
          <foo>&xxe;</foo>

Where is this a problem?

  • XML files get processed by the application
  • User Input gets written into XML (SOAP?)
  • SAML is used (XML based auth protocol for SSO)

What to do?

  • switch off XXE → XXE Prevention
  • input sanitazation (whitelisting instead of blacklisting)
  • XML Validation
  • escape all user inputs
  • to enforce: have regular code reviews
  • check your application with ZAP/Burp Suite ...

Use context aware sanitation and escaping

Insecure Deserialization

Why is this a problem?

Anytime we use user input in a serialized object or expose serialized data to the user (auth tokens)

  • HTTP cookies, HTML form parameters, API authentication tokens
  • Caching serialized objects (that contain user input)
  • Transferring Objects to other processes or over the wire (RPC/IPC)

What to do about this?

  • use integrity checks, sign serialized data and check the signature
  • logging and monitoring deserialization exceptions
  • strict type checking on deserialization (not enough - has already been circumvented)
  • deserialize in low privilege environments (when possible)
  • check your application with ZAP/Burp Suite ...

Use context aware sanitation and escaping
