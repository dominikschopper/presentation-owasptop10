Unvalidated Redirects ...

... wtf? Why is this a problem???

Where are redirects used in web applications?
How can they be exploited?

Often used to redirect after authentication
  1. You travel to https://jira.bisnode.de/browse/SOME-4711
  2. the web application sends you to the Login Form
    after filling out the data it is sent to
    https://jira.bisnode.de/login.jsp?dest=%2Fbrowse%2FSOME-4711
  3. And then you get redirected to
    https://jira.bisnode.de/browse/SOME-4711

How can unvalidated redirects be exploited?

  1. this could be sent by mail to all users in an enterprise
    https://some.web/login?redirect=http%3A%2F%2Fhack.me%2Fqwe
  2. Users click on it and get to the login page and enter their user data
  3. After logging in the redirect goes to http://hack.me/qwe
    which shows the same login page, but with the usual
    Username or Password was wrong!
  4. They enter the data again and the hacker has the Username & Password and redirect to the original login page

How can we prevent this?
 
Every Redirect: must be validated

$goto = $_GET['redirect']; // place1
$redirectMap = array(
    'place1' => $baseUrl . '/where/to',
    'place2' => $baseUrl . '/another/value',
    // ...
);
if (array_key_exists($goto, $redirectMap)) {
    $redirectUrl = $redirectMap[$goto];
} else {
    $redirectUrl = $deaultPage;
}

Cross site request forgery

... cross ... whaaat???

What is CRSF?

  • Getting an application to process data from outside the application, e.g. inserting an image in one website, that sends data to another website
  • By being in one page/application, you do something else on another page too
    The Amazon Shopping Cart is an application that allows these things
    <a href="product_affiliate_link"><img src="https://images-na.ssl-images-amazon.com/images/G/01/associates/remote-buy-box/buy1._V192207739_.gif" /></a>

Sooo, no problem with that?? If amazon allows this??

How to prevent from CSRF?

CSRF protection is especially important where the data/state of the application is changed

  • require user intaraction (enter a password again, somehow confirm, this is a real interaction)
  • use CSRF tokens in the Form (basically create a hash over the forms values and add that to a hidden field)
  • changing the state should be done via a POST request

Excursion: CSRF-Tokens

Any request that changes state should be protected

  • At session start generate a unique and large random string
  • add this token to the form 
    <form action="/transfer.do" method="post">
    <input type="hidden" name="CSRFToken" 
    value="OWY4NmQwODE4ODRjN2Q2NTlhMmZlYWE...
    wYzU1YWQwMTVhM2JmNGYxYjJiMGI4MjJjZDE1ZDZ...
    MGYwMGEwOA==">
    ...
</form>
  • validate against this token every time a form gets submitted
  • if the validation fails, inform of a possible attack

Explanation: missing function level access control

Missing function level access control?

Applications often do not protect application functions.
What should be available to an admin and not to a normal user is just hidden and not properly checked if it as called by an admin. See OWASP Juice Shop

Insecure direct Object References

Database IDs and other identifiers are used within the application without any further checks or abstraction

E.g. Show the admin overview as normal customer ...

Missing function level access control?

  • always check on that (lots of frameworks help you with that)
  • use aspect oriented feature if your language supports that
  • try to leave as few things as possible to the developer (do stuff "behind the scenes")

Missing function level access control?

  • Authorization must be checked at each call
  • Languages that support this: use Aspect oriented weaving to implement this
  • use an Express Middleware to explicitly check for this
  • use a Perl Module automatically called by Apache to check for that

Insecure direct Object References?

What can be done?

  • Use session based ids, that are only valid within the session (mapping back and forth)
  • check for access rights any time you fetch data (extra SQL)
  • prefetch valid ids and check against these

Sensitive Data Exposure?

What is Sensitive Data Exposure?

What kind of data do we speak here?

  • Usernames (or emails, passwords, ...)
  • Creditcard, banking infos, ...
  • Other systems hostnames, usernames, passwords
  • Backup Files of the application
  • Logfiles, System output, ...
  • Session IDs, Tokens ...

Sensitive Data Exposure?

How to prevent that?

Think Before You Store!
  • SSL encrypt connections
  • Webroot and public available places must not contain sensitive data (Backups, Logs, ...)
  • hash passwords
  • protect sessions, check apache config

Sensitive Data Exposure?

  • Especially with Credit Card/Banking Data - dont store it
  • Outsource this to a specialist
  • If still you need to do it: Hire an expert!
  • Scan for this using tools like OWASP Zed Attack Proxy, Nikto, W3AF, ...
  • Scan regularily, before each deploy and on productive instances also

Broken Authentication & Session Management

Broken Authentication & Session Management

  • User authentication/credentials aren’t protected when stored. See Sensitive Data Exposure.
  • Passwords, session IDs, ... are sent over unencrypted connections. See Sensitive Data Exposure.
  • Credentials can be guessed or overwritten (e.g., by weak account creation, change password, recover password, weak session IDs).
  • Session IDs are exposed in the URL (e.g., URL rewriting).
  • Session IDs don’t timeout, or single sign-on (SSO) tokens, aren’t properly invalidated during logout.
  • Session IDs aren’t rotated after successful login.

Broken Authentication & Session Management

  • hash passwords, use SSL ....
  • regenerate session IDs regularily during sessions
  • secure cookies
  • https only
  • CSRF
  • ask the password again before doing something "serious"
